# Homepage Design Benchmarks

## Research Goal

Identify best-in-class web app homepages that (1) treat the web as a first-class experience — not a gate to a mobile app download — and (2) allow users to explore or experience value before being asked to create an account. These are models to inform GoodLocal's homepage design.

**Excluded patterns (what we're not trying to emulate):**
- Strava-style: hero → "Download the app" as the primary CTA; web is subordinate
- LinkedIn-style: full account gate; every page redirects to login
- Robinhood-style: homepage is purely brand/waitlist with no product preview

---

## The Core Pattern: Progressive Value, Not a Gate

The best web app homepages share one structural principle: **earn the account creation ask by delivering value first.** They sequence the experience as:

> Understand what this is → See it work or feel the quality → Want it → Then sign up

The worst homepages invert this: the signup wall comes before any value is demonstrated. The user is asked to invest before they've been given a reason to.

## App-by-App Analysis

### 1. Duolingo — `duolingo.com`

**Hero:** "Learn a language for free. Forever."

**Why it works:**
- Headline removes the most common objections (cost, commitment) in five words
- The web experience lets you begin a lesson *before* creating an account — the product starts immediately, account creation appears at the end of the first exercise as an optional save point
- "Maybe later" is a prominent option — the app trusts you'll come back
- Massive CTA ("Get started") is not "Sign up" — it's action-framed

**Account handling:** Fully deferred. You experience ~5 minutes of product before any account prompt. Account creation is framed as "save your progress" rather than "prove you're real."

**Key takeaway:** The product *is* the onboarding. By the time you're asked for an email, you've already gotten value and feel invested.

---

### 2. Pitch — `pitch.com`

**Hero:** "Don't just present. Pitch."
**Subhead:** "The AI presentation platform that helps professionals collaborate on, create, and deliver winning slide decks — all while staying on brand."

**Why it works:**
- Headline leads with emotional differentiation ("pitch" implies confidence, persuasion) not functional feature description
- Template gallery is visible and browsable immediately in the hero — you see real outputs before signing up
- Section flow: Hero → Social proof (3M+ teams, brand logos) → How it works (4 steps) → Integrations → User-created gallery → Testimonials by role → Final CTA
- Social proof appears *before* the feature explanation — trust is established early
- "Get a demo" as secondary CTA gives high-intent visitors an alternative path

**Account handling:** Signup required to create, but browsing the public template and presentation gallery is gated only by curiosity. The gallery shows you what's possible before you commit.

**Key takeaway:** The product preview (template gallery, example presentations) is the persuasion mechanism. Copy explains; the gallery sells.

---

### 3. Luma — `lu.ma`

**Hero:** "Delightful events start here."
**Subhead:** "Set up an event page, invite friends and sell tickets. Host a memorable event today."

**Why it works:**
- Headline is emotionally resonant ("Delightful") while being immediately clear about function
- "Explore Events" is a primary nav item — visitors can browse real events without any account
- The browsable event catalog is the homepage's best product demo: real events, real communities, real use
- Primary CTA ("Create Your First Event") is action-framed and zero-friction; it assumes forward momentum

**Account handling:** Browsing is fully open. Account creation is only triggered when you want to RSVP or create an event — you can explore the full product catalog without any gate.

**Key takeaway:** The content itself (public event pages) does the selling. The homepage gets out of the way and points you toward it.

---

### 4. Linear — `linear.app`

**Hero:** Highly visual, product-screenshot-forward. Known for: "The system for modern product development."

**Why it works:**
- Opens with a visually striking, animated product screenshot — you see the interface before you read a word
- Aesthetic quality of the design communicates product quality before any copy does
- Dense feature sections that show depth — communicates "this is serious software"
- No mobile app download CTA anywhere — web app is the primary experience
- Changelog is public and discoverable without login — demonstrates active development and builds trust

**Account handling:** Browsing the marketing site and changelog is fully open. Signup prompt is persistent but not blocking — you can scroll the full page without being interrupted.

**Key takeaway:** Design quality *is* the trust signal. The visual fidelity of the homepage communicates what the product experience will feel like. For a product competing on UX quality, this is efficient signaling.

---

### 5. Canva — `canva.com`

**Hero:** "What will you design today?" (framing: starts with the user's intent, not the product's features)

**Why it works:**
- Template categories are visible immediately below the hero — Food & Drink, Social Media, Presentations, etc.
- You can click into templates and start editing *before* creating an account (account prompt appears on first save/download)
- Enormous library of starting points means every user finds something relevant within seconds
- "Start designing" CTA is output-focused — not "try the product," but "make something"

**Account handling:** Partially open. You can browse templates, open the editor, and begin working. Account creation is deferred to the save/export moment — which means you've already done the work and have a reason to complete signup ("save what you made").

**Key takeaway:** Letting users do real work before asking them to sign up is the highest-friction version of progressive disclosure — and also the most powerful. By the time you're asked, you have something to lose if you don't.

---

### 6. Figma Community — `figma.com/community`

**Why it works:**
- Community files, templates, and plugins are fully browsable without login
- You can view and inspect any public design file (view-only) without an account
- The browsable community catalog is essentially a live product demo at scale — you see real designers' work in the real tool
- Account is only needed to fork/edit/save

**Account handling:** Fully open for discovery. The community tab is a public-facing gallery that requires no authentication.

**Key takeaway:** User-generated content as a browsable public catalog is a powerful onboarding lever. The community page does what most product tours fail to do — it shows real outcomes by real people.

---

### 7. Notion — `notion.so`

**Hero:** "The connected workspace where better, faster work happens."

**Why it works:**
- Shared Notion pages are viewable by anyone without login — this means every user who shares a doc becomes an ambient product demo
- Notion's homepage shows template previews with enough visual complexity to communicate depth
- "Get Notion free" is the primary CTA — "free" removes the cost objection inline with the ask
- Teams/logo section appears just below the fold: Figma, Pixar, Nike — aspirational brand anchors

**Account handling:** Individual shared pages are fully public. The homepage requires login to access your workspace, but the template gallery and public docs are visible without account.

**Key takeaway:** Virality through shared documents is a structural onboarding mechanism — the product spreads itself. The homepage is just the entry point; the real demos are happening in the wild.

---

### 8. Airtable — `airtable.com`

**Hero:** "Build the apps that power your work" (shifted to platform framing over database framing)

**Why it works:**
- Template gallery is organized by use case (Marketing, HR, Product, etc.) — meets visitors where they are rather than forcing them to abstract
- Interactive template previews let you explore structure before committing
- Prominent "Start for free" CTA paired with "Watch a demo" — captures both ready-to-try and still-evaluating segments simultaneously
- Customer logo strip visible immediately after hero — enterprise legitimacy signal

**Account handling:** Templates are browsable. Full use requires signup, but the pre-signup exploration path is well-designed.

**Key takeaway:** Organizing the pre-signup experience around use cases (not features) is the fastest path to "this is for me." Use case recognition precedes product understanding.

## Cross-Cutting Patterns

### 1. Action-framed CTAs
The best CTAs are output-oriented, not process-oriented:
- "Start learning" not "Sign up"
- "Create your first event" not "Get started"
- "What will you design today?" not "Try Canva"
- "Start building" not "Create an account"

The CTA should answer: *what will I do*, not *what will the process require of me*.

### 2. Product visibility before the ask
Every top example makes the product tangible before the account prompt:
- Duolingo: you do a real lesson
- Canva: you open the editor
- Pitch: you see a gallery of real outputs
- Luma: you browse real events
- Figma: you inspect real files
- Linear: you see the actual interface in motion

The product is the argument. Copy sets context; the product closes.

### 3. Social proof sequencing
High-performing homepages place social proof *immediately* after the hero — not at the bottom. The credibility signal comes before the feature explanation, because it frames everything that follows.

Common social proof formats:
- Exact user/team count (Pitch: "3M+ teams"; Notion: "20M+ users")
- Recognizable brand logos (not "our customers" headers — just the logos)
- Testimonials organized by role/use case, not selected for superlative language alone

### 4. Multiple CTAs for different intent levels
Single CTA homepages leave visitors behind. The best pages offer:
- **Primary:** High-intent path ("Start for free", "Create your first event")
- **Secondary:** Evaluating path ("Watch a demo", "See how it works", "Explore templates")
- **Ambient:** Discovery path ("Browse", "Explore", "See examples")

A visitor who is not yet ready to sign up should still have somewhere useful to go.

### 5. The account creation framing matters
- Best: "Save your progress" (Duolingo) — you've already done something worth saving
- Good: "Free forever" inline with the CTA (Notion) — removes cost friction at the ask moment
- Good: "Sign up in seconds" — reduces commitment anxiety
- Weak: "Create an account" — transactional, no emotional frame
- Weak: The signup form appearing before any value has been delivered

## Implications for GoodLocal

| Pattern | GoodLocal Application |
|---|---|
| **Browse-first** | Charity listing and profiles should be fully public — no login required to discover and read about local nonprofits |
| **Action-framed CTA** | "Find a cause in Denver" or "Explore local nonprofits" — not "Sign up" or "Get started" |
| **Product in hero** | Show the actual charity discovery experience (map or list view) in the hero section |
| **Multiple intent levels** | Primary: "Explore" (no account); Secondary: "Save favorites / track giving" (requires account, deferred) |
| **Deferred account creation** | Account only becomes necessary when a user wants to track donations or save charities — not to browse |
| **Social proof placement** | Trust signals (# of Denver nonprofits listed, impact numbers, beta user quotes) immediately below the hero |
| **Use case framing** | Entry points by cause (Housing, Food, Education) meet visitors where their values already are, before explaining what GoodLocal is |

The key structural tension for GoodLocal: **Every.org handles the donation, so the actual point of account creation isn't "to donate" — it's "to remember what you've donated."** This reframes the signup ask from a permission gate into a personal record-keeping benefit. That framing ("your giving history, all in one place") should be front and center when the account prompt appears.
